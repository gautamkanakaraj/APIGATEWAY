# Architecture Brief: Multi-Tenant Serverless Function Platform ("Mini AWS Lambda")

## Overview
The primary objective of this project is to build a Serverless Function Platform that eliminates the "Complexity Tax" of traditional serverless infrastructure. It provides developers with a frictionless "Code-to-URL" deployment workflow, transforming a cloud deployment into an instant API creation tool.

---

## Core Architectural Pillars

### 1. The Control Plane (Go)
- **Status**: Implemented & Running locally.
- **Details**: Written in Go using the standard `net/http` library. Acts as the API gateway and dynamic router.
- **Endpoints**:
  - `POST /api/deploy`: Ingests code snippets and saves them to the DB.
  - `GET /user/code/:id`: Fetches code from the DB and runs it in the sandbox.
  - `GET /api/ws`: Upgrades to WebSocket to stream real-time logs to the frontend.

### 2. State Management (Neon Postgres)
- **Status**: Configured.
- **Details**: Decouples compute and storage. The database connection is handled via `pgx` / `pq` and configured using `.env` variables (`NEON_DB_URL`).
- **Schema**:
  - `functions` table: Stores function UUID, user ID, code content, public URL, and creation timestamp.

### 3. The Execution Sandbox (Lightweight Isolation)
- **Status**: Partially Implemented.
- **Details**:
  - **JavaScript Engine**: Pure-Go `goja` runner with 2-second timeout, memory isolation, and a custom `console.log` interceptor that streams logs in real-time.
  - **WebAssembly Engine**: Draft implementation of Python execution inside `wasmtime-go` using `python-3.11.wasm`.

### 4. The Frontend Interface
- **Status**: Draft Implementation.
- **Details**: A single-page application (`frontend/src/index.html`) using vanilla HTML, CSS, and JS. It features a basic code editor textarea, a deploy button, and a console log terminal that connects to the Go control plane via WebSockets.

---

## The "Code-to-URL" Lifecycle

1. **Deployment (Ingestion)**:
   A user edits code in the editor and clicks "Deploy". The browser makes a `POST` request to `/api/deploy`. The Go Control Plane generates a UUID, saves the code into Neon Postgres, and returns a public URL (`/user/code/{uuid}`).
   
2. **Triggering (Routing)**:
   When the public URL is visited, the request hits the Go Control Plane route `/user/code/:id`. The handler extracts the ID, queries Neon Postgres, and fetches the raw code content.

3. **Execution (Sandboxing)**:
   The Control Plane executes the JavaScript code in a Goja VM context, capturing stdout via a callback function.

4. **Real-Time Feedback (Streaming)**:
   While the function is executing in Goja, intercepted log lines are instantly broadcasted to active WebSocket clients (`/api/ws`) and rendered in the frontend console.

- [x] **Dynamic Router**: Ingress router implemented at `/user/code/:id` to fetch code and trigger execution.
